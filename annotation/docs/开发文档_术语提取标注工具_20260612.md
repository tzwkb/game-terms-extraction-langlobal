# 术语提取 & 标注工具 开发文档

2026-06-12
排期：预估开发 3 天（6-12 ～ 6-16），DDL 周二（6-16）
依据：需求清单（2026-06-11）、《术语标注助手》demo v2.4

## 1. 技术路径

### 1.1 总体架构：父子双工具 + 文件契约

```
原文.xlsx ──→ [父项目·AI提取工具] ──→ 候选术语.xlsx（8列模板，未审核）
                                            │ 导入合并
原文.xlsx ──────────────────────────→ [子项目·标注工具] ──→ 审定术语表.xlsx（8列模板）
                                                                  │ 回流作下次提取的术语表
                                                                  ▼
                                                          [父项目·AI提取工具]
```

- **父项目**（已有，Python + Streamlit）：本次只升级输入输出两端，管线不动；
- **子项目**（新开发）：单 HTML 文件 + 原生 JavaScript，浏览器双击即用；Excel 解析库 SheetJS 内嵌文件内，零外部依赖、完全离线；
- **契约**：8 列模板 xlsx 是两工具间唯一接口，双向读写。

### 1.2 选型理由

1. 标注的核心交互（划词、实时高亮、快捷键、行级跳转）只能在浏览器前端实现，任何后端语言都无法替代、只能附加，故子项目不引入后端；
2. Streamlit 为脚本重跑模型，适合父项目的"传参→跑批→看结果"，不适合字符级高频交互，故两工具分离；
3. 单 HTML 交付：译员零安装（无 Python 环境/无打包/无服务器），数据全程不出本机（客户文本合规）。

### 1.3 性能技术路径（子项目）

设计容量目标：**10 万行原文 × 3 万条术语全程流畅**。demo v2.4 为全量渲染方案，大文件不可用，将重写渲染与检索层：

| 技术 | 解决的问题 | 机制说明 |
|---|---|---|
| 虚拟滚动（含滚动高度比例映射） | 只渲染可视区 ~50 行，DOM 与文件大小无关；映射规避浏览器最大元素高度限制，行数可至 xlsx 格式上限 104.8 万 | 屏幕一次只看得见 30 来行，就只"摆"这几十行，滚动时把滚出去的格子换上新内容——桌上永远一屏，10 万行和 1 千行一样快。滚动条按比例造一个安全的"假高度"，像 84 层的楼配 15 格电梯面板，按比例对应楼层 |
| Aho-Corasick 多模式匹配 | 术语高亮单遍扫描，成本与词表规模无关（替代巨型正则） | 旧法拿 3 万词的名单对每句话挨个问"有你吗"；新法把词表先编成"电话簿"，句子从头读一遍顺路把所有命中词一次捞齐，耗时只看句子长短、与词表大小无关 |
| 倒排索引（术语↔行号） | 跳转定位、导出回填由全表扫描改为查表 | 导入时把"哪个词出现在哪几行"登记成册；之后点术语跳转、导出补 Key 都是直接翻册子，不再现场全表翻找 |
| 时间分片后台构建 | 建索引/建词典切片执行（约 12ms/片让出主线程），界面不冻结，旧词典服务至新词典就绪 | 浏览器里干活和画界面是同一个人，重活一口气干完界面就假死；改成切小片干，每片之间先响应你的点击，全部干完再"换班"——你看到进度提示而非白屏 |
| 输入防抖、事件委托、单元格渲染截断 | 筛选打字不卡；监听器常数个；超长单元格防 DOM 膨胀 | 防抖=停止打字 0.2 秒才算一次，不再每敲一键全表重算；事件委托=整张表一个"总管"接所有点击，代替每行雇人盯着；截断=超长格子只画前 600 字（显示本就裁 3 行），全文双击行号看 |
| localStorage 崩溃草稿 | 定时本地暂存术语库，崩溃可恢复 | 浏览器自带的本地小仓库，每分钟悄悄存一份术语库；崩溃或误关后重新打开，会问"要不要恢复刚才的草稿" |

关键算法（AC 自动机、虚拟窗口计算）已做技术预研，可行性与性能均验证通过；纯逻辑部分将配套 node 测试脚本（正确性 + 规模基准），不依赖浏览器即可回归。

### 1.4 父项目升级点（技术）

1. **统一模板导出**：管线结果增加转换函数，CLI 与 UI 同时落盘 results.xlsx（原始）与候选术语模板（8 列）；
2. **Key 列全链路携带**：加载阶段保留 行Key↔文本 映射，穿过去重与分块，按术语上下文命中行回填 ukey；
3. **列识别扩展**：表头识别统一走 LLM（自包含），同次调用识别原文列、Key 列、双语英文列；识别到英文列自动预选双语模式；
4. **回流兼容**：8 列模板可直接作术语表输入（精确表头优先，避免"术语分类"误判为术语列）。

## 2. 功能设计

### 2.1 契约：8 列模板字段

| 列 | 字段 | 父项目产出 | 子项目产出 |
|---|---|---|---|
| A | Key值（ukey） | 提取命中行的源 Key | 导出时缺失自动回填 |
| B | 术语分类 | 提取分类 | 人工修订 |
| C | 术语原文 | ✓（唯一必填） | ✓ |
| D | 术语译文 | 机器译文/官方英文照抄 | 审定译文 |
| E | 备注 | 空 | 人工备注 |
| F | 来源原文（原文定位） | 上下文句 | 来源行文本 |
| G | 审核状态 | 未审核 | 已审核/未审核 |
| H | 最新修订时间 | 提取时间 | 修订时间 |

### 2.2 子项目功能清单（对应需求 1–4）

**项目生命周期（需求 1）**：新建/手动保存/加载项目；自动保存间隔可配；工程文件采用 demo v2.4 同构 JSON（即需求所述"Trados json"），含原文+术语+列配置，加载一键复原无需二次导入，与 demo 存档双向兼容；崩溃草稿自动恢复；行号跳转与术语操作后定位。

**导入与配置（需求 2）**：原文 Excel 导入；列显隐控制；Key/原文/译文列绑定；术语表导入按模板自动识别字段（含别名容错），与 AI 提取结果合并融合——冲突提供三策略：跳过已有（默认）/仅补空白字段/覆盖；一键清空筛选、一键清空术语；快速批量添加模式（多选勾选 + Shift 连选 + 统一分类）。

**全局检索（需求 3）**：Key/原文/译文列 包含/不包含/等于；译文列另支持 为空/不为空（漏译排查）。

**术语库（需求 4）**：8 列模板导出；按审核状态/分类筛选；修订时间排序；分类标签增删（改名联动全库）；术语原文+译文搜索；逐条 审核切换/编辑/删除/点击定位（多处出现循环跳转）。

### 2.3 父项目功能清单

双语术语提取（中英两列照抄不重译）、历史文件/断点续跑、上传自动列识别、双产物下载（原始结果 + 标注模板）。

## 3. 交互方式

### 3.1 标注工具界面

左右双栏：左侧主表（原文全列表格，术语黄/蓝高亮、译文绿高亮，吸顶表头内嵌列筛选），右侧术语库（计数、搜索、筛选、分页列表、逐条操作）。顶部两条工具栏：项目操作（新建/保存/加载/自动保存/行跳转）与数据操作（导入/列绑定/清空）。

### 3.2 核心交互流

1. 导入原文 → 选 Key/原文/译文列 → 表格渲染、高亮就绪；
2. 导入候选术语（提取工具产物）→ 选合并策略 → 术语入库并全表高亮；
3. 标注：划词按 <kbd>A</kbd> / 行首 ➕ 整行 / <kbd>Q</kbd> 批量勾选回车；
4. 审核：侧栏逐条 ✓ 切换；点击术语跳转原文核对，多处出现连点循环；
5. 导出 8 列模板 → 回流提取工具。

### 3.3 快捷键

<kbd>A</kbd> 添加选中文本 ｜ <kbd>D</kbd> 删除 ｜ <kbd>E</kbd> 编辑 ｜ <kbd>Q</kbd> 批量模式 ｜ <kbd>Enter</kbd> 批量确认 ｜ <kbd>Ctrl+S</kbd> 保存 ｜ <kbd>Esc</kbd> 关闭弹窗 ｜ 双击高亮词=编辑 ｜ 双击行号=整行全文。

### 3.4 提取工具侧

上传原文+术语表 → 自动列识别（可手动改）→ 开始提取（进度条/可取消/断点续跑）→ 完成后双按钮下载：标注模板（主）+ 原始结果（辅）。

## 5. 验收标准

1. 需求清单（1–4 + 双语提取）逐条可演示；
2. 性能：10 万行 × 3 万词——导入建索引 ≤10s（后台带进度）、滚动流畅、加词/筛选/跳转 <100ms（node 基准脚本可复跑）；
3. 端到端闭环演示：提取 → 导入标注 → 审核 → 导出 → 回流提取。

## 6. 风险与对策

| 风险 | 对策 |
|---|---|
| 超大词表（>10 万）词典重建延迟 | 时间分片后台构建，界面不冻结；旧词典先行服务 |
| 浏览器差异 | 以 Chrome/Edge 为基准；滚动映射上限按最严浏览器取值 |
| Excel 解析库的外部依赖 | 直接内嵌库源码（彻底离线，单文件约 1MB） |
| 崩溃草稿受 localStorage 配额限制（约 3 万词） | 超限自动降级跳过，保留手动/定时工程备份双保险 |
