# 需求拆分方案 — 提取工具升级 vs 独立标注工具

2026-06-11
依据：PM 需求脑图、《术语标注助手 v.2.4》demo、当前仓库代码。逐项功能现状见《需求差距分析_20260611.md》。

## 拆分原则

两个工具之间只靠一份**统一 8 列模板 xlsx** 做文件契约衔接，不做进程级耦合：

- 标注工具纯前端即可成立（demo 已证明，零后端依赖），单独开发互不阻塞。
- 提取工具保持现有 Python/Streamlit 架构，只动输入输出两端。
- demo 的核心交互（划词、快捷键、单元格高亮）Streamlit 做不了，强行整合反而两边受损。

```
原文.xlsx ──→ [提取工具] ──→ 候选术语.xlsx（模板格式，未审核）
                                    │ 导入合并
原文.xlsx ──────────────────→ [标注工具] ──→ 审定术语表.xlsx（模板格式）
                                                    │ 回流作术语表输入
                                                    └────→ [提取工具] 下次运行
```

| P0 | 回流输入兼容 | 8 列模板直接作术语表输入。原模糊规则会把「术语分类」误判为中文术语列（已验证）；**已定方案：列识别统一改为 LLM 识别**——删除模糊规则，prompt 明确排除分类/Key/备注等干扰列，无 key 或失败时回退默认列并提示人工确认；可选「仅取已审核行」开关 | 小（已实现，已验证 2026-06-12） |

## B. 独立标注工具（解耦开发，基于 demo 改造）

覆盖需求**二、三、四、五全部** + 六-3 的「汇入」动作（导入模板文件并合并）。

demo 已实现（约占 80%）：项目新建/保存/加载（JSON 工程）、自动保存、原文全表展示+列显隐+列级筛选（含 ukey/原文/译文列，包含/不包含/等于）、字段绑定（key/原文/译文列）、划词/整行/批量加术语+批量分类、术语高亮与点击循环跳转定位、审核状态切换、编辑/删除/备注、分类标签管理、模糊检索、8 列模板导入导出、一键清空。

需新做（demo 缺）：

1. **合并融合冲突策略**：demo 导入遇已有术语直接跳过；需按 PM 定的策略（覆盖/保留/标记待裁决）。
2. **模板字段识别容错**：demo 按精确表头名匹配，已基本满足「绑定模板自动识别」；加别名容错即可。
3. 操作后自动跳转原文行（现为点击跳转，小增强）。
4. 防崩溃兜底：demo 自动保存是定时下载文件；建议加浏览器本地草稿（localStorage），崩溃后恢复。
5. ~~「Trados 标准 JSON」~~ **已裁决（2026-06-12）**：PM 确认即其 demo 自有的工程 JSON 格式，无关 Trados 官方格式。v3.0 工程文件已按 demo v2.4 字段名保存（originalRawData/termColumnIndex/…），双向互通：demo 可打开 v3.0 存档，v3.0 可打开 demo 存档。

## C. 衔接契约（两边唯一接口，先定稿）

| 列 | 表头（精确） | 提取工具产出 | 标注工具产出 |
|---|---|---|---|
| A | Key值 | 暂空（P1 后给真实行 Key） | 导出时回填 |
| B | 术语分类 | 提取分类 | 人工修订后 |
| C | 术语原文 | ✓ | ✓ |
| D | 术语译文 | 机器译文 | 审定译文 |
| E | 备注 | 空 | 人工备注 |
| F | 来源原文 | source_text | 来源行文本 |
| G | 审核状态 | 未审核 | 已审核/未审核 |
| H | 最新修订时间 | 提取时间 | 修订时间 |

项目 JSON 为标注工具私有格式，不跨工具。

## 解耦的代价与可选演进

「提取结果汇入术语库」从无缝单应用变为两步操作（下载→导入）。若 PM 后续坚持单一入口，可由提取工具后端托管标注前端页面（同一前端代码+本地 API），属可选演进，不影响本拆分先行。
