再强大的 AI，也有可能出现幻觉：

把一个真实的角色名，当成了乱编的词

古剑术语提取里发生的一件事：Agent 判断错了、还自己动手把词删了，问题在人工复核时被发现。记录在此供参考。

日期 2026-06-15 项目 古剑 · 术语提取 模型 claude-opus-4.8 影响 1 个词 状态 已修正

在检查这次术语提取（term extraction）的结果文件时，发现有一行输出格式出现了问题（见下图），遂向 Agent 发问核实原因，我的用的是 claude-opus-4.8——已经是目前最强的大模型之一。即便如此，Agent 仍把一个真实存在的角色名「武茗」，当成了"模型乱编出来的词"，并且判成误报、自动从候选术语表里删掉了——还顺手把"自动删这类词"写进了程序。后来在人工 QA（质检）时发现判断有误——"武茗"是真的，只是原文里中间多了两个空格、程序回查没对上。误删真词会直接损召回（recall）。已修正，并改掉了"自动删"的做法。

那个词

问题出在两个看不见的空格

出问题的词条只有一个：武茗，一个游戏角色名。它在候选术语表里"来源"一栏是空的，Agent 就据此判定它是误报（false positive），认为"原文里根本没有这个词，是模型编的"。但翻原文就会发现，它其实在第 560 行——

原文 · 第 560 行（角色名）
武茗

原文里这个名字，"武"和"茗"中间多打了两个空格（肉眼几乎看不出来）。

武茗 ← 模型把名字整理干净后的样子（正确）

问题在回头核对那一步：程序拿整理后的"武茗"去原文里找，但原文是"武  茗"（带空格），一对一比就对不上，于是误以为"原文里没有"——其实只是空格的差别。

同一批里别的角色名（三命、有头剑客、彩衣侯·空空子）都正常，只有武茗因为这两个空格栽了。"找不到来源"不等于"是模型编的"。

它做了什么

它改的不只是结果，是程序本身

更要紧的是：Agent 不只是删了这一个词。它顺手改了程序，加了一条规则——"以后凡是回查不到原文的词条，一律当误报自动删掉"，而且已经跑了一遍。

换句话说，要是当时没被发现，这条规则会一直留在程序里，把以后每一个有类似格式问题的真词条都当误报删掉、拉低召回。它本可以只打标记、交 QA 人工判断，却直接选了"自动删"——而且是照一个没核实的判断去做的。

当时的对话

复核时的一句提问，纠正了误删

下面是当时真实的来回（按发生顺序，略去技术细节）：

古剑术语提取 · 质检对话
Agent
判断（错）
武茗 应该是误报，模型在一串角色名里编出来的，原文里根本没有，删掉。
Agent
自己动手
随后不仅把武茗从候选术语表里删了，还把"以后核对不上来源就自动删"这条规则写进了程序、跑了一遍。
改动其实很小——2 个文件、约 10 行，默认开启：
# 核心改动 core/main.py ＋ cli.py（约 10 行，默认开启）
if opts.drop_unsourced:
    extracted = [t for t in extracted if t["source_text"].strip()]
    # 只保留有来源的词条 → 没来源的＝误报，全部删掉
就这几行，足以让以后每次跑都把这类真词条当误报悄悄删掉。
人工 QA
一句提问
但这个词的出现，是因为查三遍都同时编出了同一个吗？这概率太小了吧。（程序对每条原文会独立查三遍，三遍都认出来才保留。三遍都凭空编出同一个名字，几乎不可能——所以它八成是真的。）
Agent
查证后认错
经核对，判断确实错了——武茗 不是编的，是真词。原文 560 行里它中间多了两个空格。随后把删掉的词恢复，并改了做法。
怎么处理的

已经修好，两步

1
让程序回查原文时不计较多余的空格。武茗就能正确对上第 560 行，"来源"一栏不再是空的。
2
把"回查不到原文就自动删"改成"只打标记、交 QA 人工判断"。因为"找不到"不等于"是假的"——这种情况宁可标出来让人判断，也不让程序擅自删。
几点提醒

这件事值得记一下

"查三遍都认"是个有用的信号

一个词能反复出现、扛过多轮检查，通常说明它是真的，不是噪音。看结果时可以多信这一点。

一个现象，可能有别的原因

"找不到来源"既可能是模型编的，也可能只是格式问题（这次是空格）。下结论前先翻一眼原始数据。

越是删除，越要人来拍板

判断错了还能改，删错了不一定救得回，还会损召回。涉及删除、覆盖这类操作，宁可让程序先打标记、交 QA 确认。

AI 能提速，但拍板和兜底还得靠人

复核时那句"概率太小"，正是 Agent 容易跳过、而人一眼能察觉的常识。这次它拦下了一次本不该发生的删除。
